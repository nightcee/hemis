Technical Portability Analysis

Hemis PTX on SUI
Blockchain

A deep-dive into whether Hemis and its Probabilistic Transaction (PTX) feature can be ported to the SUI blockchain.

01 / Overview

What is Hemis?

Hemis is an open-source, community-driven cryptocurrency built on top of Bitcoin Core in C++, using the UTXO model. Its key distinguishing features are:

  • Proof of Stake consensus mechanism
  • Gamemaster Nodes — analogous to Dash Masternodes, running LLMQ (Long-Living Masternode Quorum) to produce a VRF (Verifiable Random Function)
  • PTX (Probabilistic Transactions) — on-chain probabilistic transactions designed for gaming and wagering use cases
02 / PTX Architecture

How PTX Works

A Probabilistic Transaction (PTX) is an extension of the existing CTransaction that encodes a list of possible outputs along with a probability range for each. When the PTX is included in a block, the network uses the block's Gamemaster VRF output combined with the transaction ID to derive a random number. That number determines the winning output; all other outputs are discarded. The entire mechanism runs at the consensus layer — the game outcome does not depend on any off-chain secrets or external oracle.

04 / Component Analysis

PTX Components on SUI — Feasibility Per Layer

Portable 1. Probabilistic Output Structure (CProbabilisticTransaction)

In Hemis, this is a direct modification to the transaction serialization format — a protocol-layer change. On SUI, Move structs can model the same "RangeOutput" logic, but it lives at the contract layer, not the consensus layer. The concept is portable; the implementation location is entirely different.

Portable 2. VRF Randomness

Hemis derives a random number as r = Truncate32(Hash(ptx.GetHash(), vrf)), storing the VRF value in each block header so miners cannot predict or manipulate results. SUI's native sui::random module provides equivalent on-chain VRF functionality — and is actually easier to integrate at the application layer. This is the most straightforward component to port.

Portable 3. Seed Height / Expiry Height (nSeedHeight / nExpiryHeight)

Hemis enforces through mempool rules that the seed height is not in the past and not too far in the future (e.g., within 10 blocks). On SUI, the Clock object or epoch numbers can simulate these time bounds. Portable, though with slightly different semantics.

Not Portable 4. Gamemaster Node Network

This is the soul of Hemis PTX. Activating PTX is a consensus change requiring a supermajority of Gamemasters to signal readiness before PTX rules are enforced, preventing accidental chain splits. SUI has no equivalent Masternode concept — its validators are a permissioned PoS set. This layer cannot be directly ported; a multisig committee or DAO governance contract would be the closest substitute.

Partial 5. House Edge Enforcement (≤ 1%)

Hemis enforces this at the consensus layer: the total outputs must not exceed inputs minus fees, and the house edge is capped (e.g., ≤ 1%). On SUI this can be implemented in contract logic, but it is not protocol-level enforcement — its security depends entirely on the contract's own design. The guarantee is weaker.

05 / Migration Path

Building PTX Concepts on SUI

If you genuinely want to implement PTX-equivalent functionality on SUI, the most viable architecture looks like this:

// SUI Move Contract Architecture
module hemis_ptx::game {

  // Bet object — holds stake and probability ranges
  struct GamePool has key, store {
    id: UID,
    stake: Balance<HMS>,
    ranges: vector<RangeOutput>,
    seed_epoch: u64,
    expiry_epoch: u64,
  }

  // Resolve outcome using sui::random (on-chain VRF)
  public fun resolve(pool: GamePool, rand: &Random, ctx: &mut TxContext) {
    let r = random::generate_u32(rand);
    // locate winning RangeOutput and transfer funds
  }

  // DAO governance — replaces Gamemaster supermajority
  struct GovernanceProposal has key { ... }
}

SUI Advantages in This Context

  • sui::random is easier to integrate than Hemis LLMQ VRF at the application layer
  • Move's object ownership model naturally fits the lifecycle of a "bet object"
  • High TPS and low fees are far better suited for gaming workloads
  • No need to run full nodes / Masternode infrastructure

What Is Lost

  • Consensus-layer-enforced fairness (Hemis PTX's biggest selling point)
  • The Gamemaster decentralized network trust model
  • Protocol-level house edge caps — contract bugs can bypass them
06 / Conclusion

Final Verdict

Concept: ✅ Yes  |  Direct Port: ❌ No

The core idea of Hemis PTX — on-chain verifiably fair random gaming — can absolutely be realized on SUI. However, this is not a "port"; it is a complete redesign.

Hemis's uniqueness lies in embedding betting logic into the consensus layer, while SUI's philosophy places logic in the contract layer. The two differ fundamentally in their trust assumptions and attack surfaces.

If your goal is to build a decentralized gaming platform on SUI with similar functionality: SUI's sui::random + Move contracts are entirely sufficient, and development is considerably easier.

If your goal is to replicate Hemis's "consensus-layer-enforced fairness" security model: SUI's current architecture cannot achieve this.
